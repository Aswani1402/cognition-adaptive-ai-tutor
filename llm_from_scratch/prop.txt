PHASE 3 — GENERATE RICH LEARNING PACKETS

Create: scripts/generate_teaching_aligned_packets.py

This is the MOST IMPORTANT script. Read this carefully.

CONTENT RULE:
Every single field in every packet MUST use the ACTUAL TEXT
fetched from concept_resources. Do not produce placeholder text.
Do not produce one-line content. Do not write "..." or "N/A".
If a field is missing from the DB, derive it from base_content.

For every concept, for every teaching_view, generate a full packet.
That means 38 concepts × 14 teaching_views = 532 packets minimum.

Each packet JSON must follow this exact structure:

{
  "packet_id": "<domain>_<concept_id>_<teaching_view>",
  "domain": "Python",
  "concept_id": "P1",
  "concept_name": "Variables",
  "difficulty": "easy",
  "teaching_view": "definition_view",

  "teaching_content": {
    "title": "<concept_name> — <teaching_view label>",

    "beginner_explanation": "
      A full paragraph (at least 4 sentences) explaining this concept
      at a beginner level using real words from base_content.
      No bullet points here. Full natural language explanation.
      Example: 'In Python, a variable is like a labelled box that stores
      a value. When you write x = 5, you are telling Python to create
      a box named x and put the number 5 inside it. You can open that
      box later, change what is inside, or use the value in a calculation.
      Variables make your code flexible because the same logic works with
      different values just by changing what the variable holds.'
    ",

    "definition": "
      The exact definition from the DB base_content field, expanded
      into 2-3 full sentences if the DB value is short.
      Must not be one line only.
    ",

    "why_it_matters": "
      2-3 sentences from real_world_use DB field explaining
      why this concept matters in practice.
    ",

    "step_by_step": [
      "Step 1: <concrete step using concept text>",
      "Step 2: <concrete step>",
      "Step 3: <concrete step>",
      "Step 4: verify or use the result"
    ],

    "example": "
      A concrete example using actual examples from the DB examples field.
      If it is code, write the code block as a string with newlines.
      If it is non-code, write a full sentence example.
      Must be a real example, not generic placeholder.
    ",

    "code_or_task_example": "
      A short runnable code snippet (for code-capable concepts)
      OR a written task description (for non-code concepts).
      Derived from DB examples field.
    ",

    "common_mistake": "
      One real misconception from the DB misconceptions field,
      written as: 'Many learners think X but actually Y because Z.'
      Must use the actual misconception text from the DB.
    ",

    "real_world_use": "
      1-2 sentences from DB real_world_use field.
      Must be specific to this concept, not generic.
    ",

    "key_points_used": [
      "<key point 1 from DB key_points field>",
      "<key point 2 from DB key_points field>",
      "<key point 3 if available>"
    ],

    "misconceptions_used": [
      "<misconception 1 from DB misconceptions field>",
      "<misconception 2 if available>"
    ],

    "quick_check": "
      One question the learner can ask themselves to verify
      they understood this teaching view.
      Example: 'Can you write a variable assignment from memory
      without looking at the notes?'
    ",

    "revision_line": "
      One summary sentence that captures the core idea.
      Example: 'A variable stores a named value that can be
      read and changed throughout your program.'
    "
  },

  "aligned_assessments": [
    {
      "task_type": "mcq",
      "question": "<question that directly tests what was taught in teaching_content>",
      "options": [
        "A) <correct answer derived from teaching_content>",
        "B) <plausible distractor from misconceptions>",
        "C) <plausible distractor>",
        "D) <plausible distractor>"
      ],
      "answer": "A",
      "explanation": "
        2-3 sentences explaining why A is correct
        and why the distractors are wrong.
        Must reference the teaching_content taught above.
      ",
      "alignment_reason": "
        This MCQ tests the exact definition shown in definition_view.
        The distractors are derived from the concept's known misconceptions.
      "
    },
    {
      "task_type": "fill_in_the_blank",
      "question": "<sentence with one blank using concept vocabulary>",
      "answer": "<exact word or phrase that fills the blank>",
      "explanation": "<why this answer is correct>",
      "alignment_reason": "Tests recall of the key term introduced in this view."
    }
  ],

  "hint": "
    A useful hint that guides without giving the full answer.
    Must be specific to this concept and teaching_view.
    Not a generic 'think about it' line.
    Example: 'Remember that a variable name cannot start with a number.
    Try writing the assignment again and check if the name is valid.'
  ",

  "feedback_template": {
    "correct": "
      Positive feedback + one extension thought.
      Example: 'Correct! You understood that x = 5 assigns the integer 5
      to variable x. Now try changing x to a string and see what happens.'
    ",
    "partial": "
      Acknowledge what they got right + what to review.
      Example: 'You got the assignment syntax right but mixed up
      integer and string types. Go back to the type section above.'
    ",
    "wrong": "
      Non-discouraging correction + specific pointer back to content.
      Example: 'Not quite. Remember that = in Python means assignment,
      not equality. Re-read the definition section above.'
    "
  },

  "revision_summary": "
    3-5 sentences summarising everything taught in this packet.
    Must cover: definition, example, common mistake, real-world use.
    Written as a compact review paragraph, not bullet points.
  ",

  "next_step": "
    One sentence recommending what the learner should do next.
    Must reference next_concept_link from DB if available.
    Example: 'Now that you understand Variables, move on to
    Data Types to learn what kinds of values a variable can hold.'
  "
}

TEACHING VIEW RULES:
Each teaching_view must produce different teaching_content emphasis:

definition_view:
  - beginner_explanation focuses on: what is this concept, plain English definition
  - aligned_assessments: mcq testing definition, fill_in_the_blank on key term
  - difficulty: easy

simple_example_view:
  - beginner_explanation focuses on: walk through the example step by step
  - Use the actual example from DB examples field as the centrepiece
  - aligned_assessments: output_prediction or mcq based on that exact example
  - difficulty: easy

step_by_step_view:
  - beginner_explanation focuses on: numbered steps to use this concept
  - step_by_step list must have 4+ actual steps derived from DB content
  - aligned_assessments: order_the_steps or mcq about which step comes first
  - difficulty: easy-medium

analogy_view:
  - beginner_explanation focuses on: a real-life analogy for this concept
  - The analogy must be specific, not generic (not just 'like a box')
  - aligned_assessments: analogy_check mcq
  - difficulty: easy

code_view:
  - beginner_explanation focuses on: reading and writing the actual code
  - code_or_task_example must contain a runnable code block from DB
  - aligned_assessments: output_prediction, code_completion_task
  - difficulty: medium

debug_view:
  - beginner_explanation focuses on: what goes wrong with this concept and why
  - common_mistake must come from DB misconceptions field
  - code_or_task_example must show a BUGGY version then the FIXED version
  - aligned_assessments: debug_task, spot_the_error
  - difficulty: medium

output_prediction_view:
  - beginner_explanation focuses on: tracing through code/logic to predict output
  - code_or_task_example must show code and ask what happens when run
  - aligned_assessments: output_prediction with exact answer from DB examples
  - difficulty: medium

misconception_view:
  - beginner_explanation focuses on: correcting the most common wrong belief
  - Use EVERY misconception from DB misconceptions field
  - aligned_assessments: true_or_false on the misconception, mcq where wrong option is the misconception
  - difficulty: medium

transfer_view:
  - beginner_explanation focuses on: how this concept applies in a different context
  - Use real_world_use from DB as the transfer scenario
  - aligned_assessments: transfer_question, short_answer
  - difficulty: hard

challenge_view:
  - beginner_explanation focuses on: pushing beyond basics to a harder use case
  - aligned_assessments: challenge_question, multi_step_challenge
  - difficulty: hard

revision_view:
  - teaching_content is a revision of ALL the above views
  - Must include definition + example + mistake + real-world use in one packet
  - aligned_assessments: mcq from each sub-topic, flashcard_concept
  - difficulty: revision

flashcard_view:
  - teaching_content is condensed to front/back format
  - Generate 5 flashcards per concept: definition, syntax, example, mistake, real-world
  - Each flashcard: { "front": "...", "back": "..." }
  - aligned_assessments: concept_recall, keyword_match
  - difficulty: revision

mindmap_view:
  - teaching_content describes the concept as a mindmap structure
  - mindmap field: { "centre": "concept_name", "branches": [ { "label": "...", "children": [...] } ] }
  - Branches: Definition, Examples, Key Points, Common Mistakes, Real-World Use, Next Steps
  - Each branch must have 2-4 children from DB content
  - difficulty: revision

voice_script_view:
  - teaching_content is written as a natural spoken script
  - beginner_explanation must read like a teacher speaking out loud
  - No bullet points in this view. Full spoken paragraphs only.
  - Example style: "Today we are going to talk about Variables in Python.
    Think of a variable like a named box. You put something in the box,
    give it a name, and any time you need it, you just call the name..."
  - aligned_assessments: quick_check spoken as a question
  - difficulty: easy-medium
